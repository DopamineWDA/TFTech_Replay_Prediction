# IFTech 算法笔试题分析报告

## 1. 任务目标与解题思路

本题的目标是：基于用户首次播放播客单集时的行为上下文、用户画像信息和单集侧静态信息，预测该用户在未来一个月内是否会对该单集发生重复播放行为，并输出测试集上每条样本的概率值。

这是一个大规模结构化数据上的二分类概率预测问题。题目不仅要求给出 `result.csv`，还要求体现：

- 特征工程与数据理解能力
- 模型选择与训练策略
- 数据分析与报告能力
- 对 AI 工具的合理使用

因此，本次提交并不只是完成一版可运行代码，而是围绕完整答题链路展开：

1. 先做 EDA，明确哪些字段有信息量、哪些字段存在缺失与异常、训练集与测试集是否有分布差异。
2. 基于 EDA 结论设计一套可解释、低泄漏风险、能够在大规模数据上稳定运行的结构化特征。
3. 选择适合大样本表格任务的 LightGBM 进行训练，并使用交叉验证控制泛化风险。
4. 输出预测结果、交叉验证指标、OOF 预测和特征重要性，并在报告中对结果进行解释。

## 2. 数据概览与问题理解

本次建模使用了 5 张表：

- `train.csv`
- `test.csv`
- `user_feature.csv`
- `episode_feature.csv`
- `episode_additional.csv`

数据规模如下：

- `train.csv`：`7,377,418` 行
- `test.csv`：`2,556,790` 行
- `user_feature.csv`：`34,403` 行
- `episode_feature.csv`：`2,296,833` 行
- `episode_additional.csv`：`2,296,869` 行

训练集标签分布如下：

- `label = 1`：`3,714,656`，占比 `50.35%`
- `label = 0`：`3,662,762`，占比 `49.65%`

从标签分布可以看出，这不是一个极端不平衡问题，因此本次工作重点不在采样，而在于：

- 识别强信号类别字段
- 构造稳定的统计类特征
- 处理缺失与异常
- 避免 target leakage
- 在训练集与测试集分布不完全一致的前提下维持泛化能力

## 3. 数据理解与 EDA

### 3.1 主表结构与业务直觉

`train.csv` / `test.csv` 中包含 5 个核心字段：

- `uid`
- `episode_id`
- `tab_name`
- `scene_name`
- `entrance_type`

其中：

- `uid` 标识用户
- `episode_id` 标识单集
- `tab_name`、`scene_name`、`entrance_type` 描述用户首次播放行为发生时的上下文环境

从业务直觉上看，重复播放行为并不只由“用户喜欢什么内容”决定，也受到“用户是从什么入口进入内容”的影响。比如主动收藏、列表播放、推荐入口、搜索入口对应的复听意愿可能不同。因此，主表中的上下文字段不是噪声，而是重要信号。

### 3.2 数据规模特征

训练集存在明显的用户重复与单集重复：

- 训练集唯一用户数：`30,755`
- 训练集唯一单集数：`359,966`
- 平均每个用户样本数：`239.88`
- 平均每个单集样本数：`20.49`

这意味着：

- 同一用户在训练集中出现大量重复，用户历史统计天然有价值。
- 单集也有重复，但长尾明显，因此单集热度和单集历史表现不能忽略。
- 由于同一 `uid` 和 `episode_id` 会大量重复出现，直接使用这些高基数 ID 做类别编码存在拟合风险，更适合转为频次统计和 target encoding。

### 3.3 缺失情况与处理思路

训练集缺失情况：

- `scene_name` 缺失 `37.58%`
- `entrance_type` 缺失 `6.19%`
- `tab_name` 缺失 `0.34%`

测试集缺失情况：

- `scene_name` 缺失 `45.05%`
- `entrance_type` 缺失 `7.50%`
- `tab_name` 缺失 `0.33%`

用户表中：

- `sex` 缺失 `19,902 / 34,403`

单集表中：

- `category_ids` 存在一定缺失
- `language` 几乎无缺失

我对缺失值采取的处理原则是“保留并显式编码”，而不是删除样本。原因如下：

- `scene_name` 缺失比例很高，删行会损失大量训练数据。
- 在推荐或内容消费问题里，“缺失本身”往往就代表一种行为状态，例如某些入口链路不会产出该字段。
- 对于树模型而言，将缺失作为单独类别保留，通常比粗暴填均值或删样本更稳健。

因此，本次特征工程中统一将类别缺失填充为 `MISSING`。

### 3.4 类别分布与标签关系

EDA 显示，`tab_name`、`scene_name`、`entrance_type` 与正样本率关系非常明显。

以 `tab_name` 为例：

- `collections`：占比 `49.95%`，正样本率 `0.6197`
- `discover_page`：占比 `29.54%`，正样本率 `0.4158`
- `search`：占比 `8.45%`，正样本率 `0.4214`
- `personal_list`：占比 `6.46%`，正样本率 `0.2227`

以 `scene_name` 为例：

- `F`：占比 `43.76%`，正样本率 `0.6370`
- 缺失值：占比 `37.58%`，正样本率 `0.3865`
- `A1`：占比 `5.70%`，正样本率 `0.3906`
- `D1`：占比 `2.90%`，正样本率 `0.5170`
- `G`：占比 `1.03%`，正样本率 `0.6572`

以 `entrance_type` 为例：

- `collections`：占比 `30.65%`，正样本率 `0.6321`
- `guess`：占比 `26.67%`，正样本率 `0.4250`
- `play_list`：占比 `14.63%`，正样本率 `0.6577`
- `anything`：占比 `6.55%`，正样本率 `0.2197`

这组结果说明了两点：

- 入口和场景字段是本题最核心的信号来源之一。
- 这些字段不能只做“是否缺失”的浅层处理，而应保留原始类别，并进一步构造交叉统计特征。

### 3.5 用户特征理解

用户侧 EDA 得到几个比较关键的观察：

- 所有训练、测试用户都能在 `user_feature.csv` 中找到。
- `age = 0` 占比约 `57.94%`，更像“未知年龄编码”而不是真实年龄。
- `age` 最大值达到 `1052`，存在明显异常值。
- `membership_days` 出现负值，说明 `rg_date` 与 `exp_date` 存在异常记录。

从标签关系上看：

- 年龄分桶存在一定区分度
- `rg_source` 对正样本率也存在差异

这些现象决定了用户侧特征不能只把 `age` 当普通连续值直接喂给模型，而要做：

- 原值保留
- 年龄分桶
- `age=0` 单独标记
- 异常年龄标记
- 会员天数异常标记

这也是本次特征工程中加入 `age_bucket`、`is_age_zero`、`is_age_outlier`、`is_membership_days_negative` 的原因。

### 3.6 单集特征理解

单集侧 EDA 的结论主要包括：

- `duration` 中位数约 `226,626 ms`，约 `3.78` 分钟
- 大多数样本只有 `1` 个类目、`1` 个主播、`1` 个制作人、`1` 个作者
- `language` 更像编码字段而不是自然语言文本

按时长桶看正样本率：

- `<2m`：`0.3985`
- `2-3m`：`0.4495`
- `3-4m`：`0.5120`
- `4-5m`：`0.5112`
- `5-10m`：`0.4741`
- `10m+`：`0.3768`

这表明 `duration` 与标签并非线性关系，而更像区间关系：中等时长内容重复播放率更高，极短和极长内容相对较低。因此我没有只保留原始时长，而是同时构造：

- `duration_ms`
- `duration_sec`
- `duration_log1p`
- `duration_bucket`

多值字段 `category_ids`、`host`、`producer`、`writer` 在 baseline 中没有做完整展开，而是先使用数量统计：

- `category_count`
- `host_count`
- `producer_count`
- `writer_count`

这样做的原因是，当前阶段更关注稳健性和可解释性，避免在第一版中引入过高维度和更重的内存开销。

### 3.7 附加单集信息理解

`episode_additional.csv` 中最有价值的两个字段是：

- `title`
- `uuid`

EDA 发现：

- `title` 更像脱敏 token 串，不适合直接进行语义 NLP 建模
- `uuid` 唯一值很多，但约 `10.32%` 的 `uuid` 存在共享情况

因此本次处理方式是：

- `title` 只提取浅层统计特征：`title_token_count`、`title_char_len`
- `uuid` 保留原值，并计算 `uuid_episode_count`

`uuid_episode_count` 可以反映同一身份标识对应多个单集的情况，这对识别重复发布、系列内容或内容复用具有一定价值。

### 3.8 训练 / 测试分布漂移

训练集和测试集在多个字段上存在明显漂移。

`tab_name`：

- `collections`：训练 `49.95%`，测试 `39.87%`
- `discover_page`：训练 `29.54%`，测试 `34.07%`

`scene_name`：

- `F`：训练 `43.76%`，测试 `33.05%`
- 缺失值：训练 `37.58%`，测试 `45.05%`

`entrance_type`：

- `collections`：训练 `30.65%`，测试 `22.78%`
- `guess`：训练 `26.67%`，测试 `30.29%`

这说明本题不适合单纯追求训练集拟合，而需要更关注泛化。也正因为如此，我在建模时强调：

- 使用交叉验证而不是单次切分
- 避免泄漏型高风险特征
- 使用结构化树模型做稳健建模
- 通过 OOF target encoding 保留统计信息但避免在训练集上“看答案”

## 4. 特征工程设计与实现

本次特征工程围绕“行为上下文 + 用户画像 + 单集画像 + 统计特征 + 不泄漏监督统计”五个层次展开。

### 4.1 原始行为上下文特征

直接保留主表中的：

- `uid`
- `episode_id`
- `tab_name`
- `scene_name`
- `entrance_type`

其中，`tab_name`、`scene_name`、`entrance_type` 作为显式类别特征输入模型；`uid`、`episode_id` 由于高基数和较强过拟合风险，不直接作为 GPU LightGBM 的类别特征使用，而主要通过统计特征表达其信息。

### 4.2 用户侧静态特征

来自 `user_feature.csv` 的特征包括：

- `address`
- `sex`
- `rg_source`
- `age`
- `age_bucket`
- `rg_year`
- `rg_month`
- `exp_year`
- `exp_month`
- `membership_days`
- `is_age_zero`
- `is_age_outlier`
- `is_membership_days_negative`

设计亮点在于：我没有简单删掉异常数据，而是把异常转换为模型可学习的信息。这种处理方式比“只做清洗”更符合真实业务场景，因为异常记录本身可能反映了用户画像质量、历史系统记录方式或渠道差异。

### 4.3 单集侧静态特征

来自 `episode_feature.csv` 与 `episode_additional.csv` 的特征包括：

- `language`
- `duration_ms`
- `duration_sec`
- `duration_log1p`
- `duration_bucket`
- `category_count`
- `host_count`
- `producer_count`
- `writer_count`
- `title_token_count`
- `title_char_len`
- `uuid`
- `uuid_episode_count`

这些特征的作用分别对应：

- 内容长度特征
- 内容结构复杂度特征
- 内容身份复用特征
- 标题浅层结构特征

### 4.4 无监督频次统计特征

为表达“用户活跃度”“单集热度”“行为组合熟悉度”，我构造了以下频次特征：

单字段频次：

- `uid_count`
- `episode_count`
- `tab_name_count`
- `scene_name_count`
- `entrance_type_count`

交叉频次：

- `uid_tab_count`
- `uid_entrance_count`
- `episode_tab_count`
- `episode_entrance_count`

这部分不依赖标签，因此不存在泄漏风险，但对模型非常有用。例如：

- `uid_count` 反映用户历史活跃程度
- `episode_count` 反映单集热度
- `uid_entrance_count` 反映用户是否经常通过某个入口发生播放行为

### 4.5 OOF target encoding

考虑到主表中的类别字段和高基数 ID 都与标签存在较强关系，我进一步设计了 OOF target encoding 特征：

单字段 target encoding：

- `te_uid`
- `te_episode_id`
- `te_tab_name`
- `te_scene_name`
- `te_entrance_type`

交叉字段 target encoding：

- `te_uid_tab`
- `te_uid_entrance`
- `te_episode_tab`
- `te_episode_entrance`

辅助字段：

- `te_*_train_count`
- `global_label_mean`

这部分是本次特征工程的核心亮点之一。原因在于：

- 直接用全训练集标签均值回填训练样本会发生 target leakage。
- 这类泄漏在高基数字段上尤其危险，因为稀有分组可能几乎等于把答案直接告诉模型。

因此，我没有采用全量 target encoding，而是明确使用 OOF 方式生成训练特征。

### 4.6 泄漏控制实现

本次 OOF target encoding 的实现逻辑是：

1. 先基于 `uid + episode_id + row_idx` 的稳定哈希，将训练样本分到固定 fold。
2. 统计每个分组在全体训练集上的标签和与样本数。
3. 同时统计每个分组在当前 fold 内的标签和与样本数。
4. 对训练集中的每条样本，使用“全体统计 - 本 fold 统计”得到当前样本可用的 target encoding。
5. 对测试集中的每条样本，使用全训练集统计结果生成编码。

这样做的好处是：

- 每条训练样本不会看到自己所在 fold 的标签信息
- 实现上可以流式统计，适合大文件
- 在大规模结构化数据上比复杂深度特征更稳健

### 4.7 特征工程的工作亮点

本次特征工程最能体现个人工作的部分主要有四点：

- 没有只停留在字段拼接，而是将缺失、异常、长尾、高基数和分布漂移都纳入特征设计。
- 对用户、单集和上下文字段做了分层建模，而不是只关注某一侧信息。
- 在大规模数据上实现了工程可运行的 OOF target encoding，控制了标签泄漏。
- 特征设计既保留了可解释性，也兼顾了训练成本，适合作为正式提交的 baseline。

## 5. 模型选择与训练策略

### 5.1 模型选择

本次提交选择 `LightGBM` 作为主模型，原因如下：

- 适合大规模结构化数据二分类任务
- 对数值特征和类别特征兼容性较好
- 训练速度快，适合数百万级样本
- 可以直接输出概率
- 便于输出特征重要性，支撑后续分析

我没有在本次提交中优先使用深度模型或文本模型，主要是因为：

- 当前特征主体仍是表格结构化数据
- `title` 不具备高质量自然语言语义特征
- 大部分强信号来自上下文类别、用户统计和单集统计
- 在笔试场景中，可解释、稳定、可复现的树模型更能体现工程取舍能力

### 5.2 模型输入

模型训练使用的是特征工程脚本生成的两张特征表：

- `IFTech/feature/train_features_baseline.csv`
- `IFTech/feature/test_features_baseline.csv`

训练集中额外包含：

- `row_id`
- `label`

测试集中额外包含：

- `id`

### 5.3 类别特征处理

作为 `LightGBM categorical_feature` 输入的列包括：

- `tab_name`
- `scene_name`
- `entrance_type`
- `address`
- `sex`
- `rg_source`
- `age_bucket`
- `language`
- `duration_bucket`

原始高基数 ID：

- `uid`
- `episode_id`
- `uuid`

默认不直接作为模型输入特征使用。这是一个刻意的策略，而不是遗漏。核心考虑是：

- GPU LightGBM 对超高基数类别的处理效率和稳定性不如中低基数类别
- 直接输入高基数 ID 可能导致模型记忆训练样本而非学习规律
- 这些字段的信息已经通过频次特征和 OOF target encoding 得到保留

### 5.4 验证策略

最终使用 `3 fold StratifiedKFold`。

理由如下：

- 数据量已经很大，3 折仍有较高代表性
- 相比 5 折，训练耗时更可控
- 对笔试场景而言，3 折已经足以判断结果稳定性

这里的关键不是“折数越大越好”，而是在计算成本与验证稳定性之间做取舍。对于数百万级样本，本次选择更加偏向工程效率和稳定落地。

### 5.5 训练参数

本次提交结果对应的主要参数如下：

- `device = gpu`
- `gpu_id = 0`
- `n_splits = 3`
- `num_boost_round = 1500`
- `early_stopping_rounds = 100`
- `learning_rate = 0.03`
- `num_leaves = 255`
- `feature_fraction = 0.8`
- `bagging_fraction = 0.8`
- `bagging_freq = 1`
- `min_data_in_leaf = 100`
- `max_bin = 255`
- `deterministic = True`
- `force_col_wise = True`

参数风格整体偏保守，核心目标是：

- 保持训练稳定
- 控制泛化风险
- 在数百万样本上实现较好的收敛表现

### 5.6 训练过程理解

训练监控结果显示，三折模型都跑到了 `1500` 轮上限，未触发 early stopping：

- Fold 0：`best_iteration = 1500`
- Fold 1：`best_iteration = 1500`
- Fold 2：`best_iteration = 1500`

这说明：

- 当前参数设置没有出现明显过拟合拐点
- 模型仍有继续提升空间
- 但考虑到训练成本和提交时效，本次提交版本选择先固定在这组稳定参数上

这个取舍同样体现了本次工作的思路：先交付一版完整、稳定、可解释、可复现的高质量 baseline，而不是为了追求边际提升把时间全部投入到过度调参。

## 6. 训练结果与结果分析

### 6.1 交叉验证结果

本次训练结果输出在：

- `IFTech/lgbm_runs/run_seed42_gpu_output`

整体交叉验证结果：

- `CV AUC = 0.819658`
- `CV Logloss = 0.517444`

各折结果：

- Fold 0：`AUC = 0.819546`，`Logloss = 0.517598`，`best_iteration = 1500`
- Fold 1：`AUC = 0.819251`，`Logloss = 0.517868`，`best_iteration = 1500`
- Fold 2：`AUC = 0.820180`，`Logloss = 0.516865`，`best_iteration = 1500`

从结果上看：

- 三折 AUC 非常接近，说明模型在不同数据切分下表现稳定。
- Logloss 也比较集中，说明概率输出没有明显失稳。
- 整体结果表明，这套特征工程和训练策略在泛化层面是成立的。

### 6.2 训练结果的解释

`AUC` 反映模型区分正负样本的排序能力，`0.819658` 对于没有引入复杂文本建模、以结构化特征为主的 baseline 来说已经是较有竞争力的结果。

`Logloss` 反映概率输出质量，`0.517444` 说明模型不仅能排序，也能给出相对合理的概率值。由于题目提交的是概率文件，因此这一点很重要。

### 6.5 结果分析对比

本次结果分析中的“对比”主要体现在三个层面。
