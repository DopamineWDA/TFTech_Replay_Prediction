# IFTech EDA 摘要

## 1. 数据规模

- `train.csv`：`7,377,418` 行
- `test.csv`：`2,556,790` 行
- `user_feature.csv`：`34,403` 行
- `episode_feature.csv`：`2,296,833` 行
- `episode_additional.csv`：`2,296,869` 行

## 2. 标签分布

- `label=1`：`3,714,656`，占比 `50.35%`
- `label=0`：`3,662,762`，占比 `49.65%`

结论：标签基本均衡，本题主要矛盾不在类别不平衡，而在特征表达和泛化能力。

## 3. 主表基础观察

- 训练集唯一用户数：`30,755`
- 训练集唯一单集数：`359,966`
- 平均每个用户样本数：`239.88`
- 平均每个单集样本数：`20.49`

结论：

- 用户在训练集中重复出现非常明显，用户历史统计特征价值高。
- 单集具有长尾分布，热门单集与长尾单集需要同时兼顾。

## 4. 缺失情况

训练集：

- `scene_name` 缺失 `37.58%`
- `entrance_type` 缺失 `6.19%`
- `tab_name` 缺失 `0.34%`

测试集：

- `scene_name` 缺失 `45.05%`
- `entrance_type` 缺失 `7.50%`
- `tab_name` 缺失 `0.33%`

用户表：

- `sex` 缺失 `19,902 / 34,403`

单集表：

- `category_ids` 缺失 `94,145`
- `language` 缺失 `4`

结论：

- `scene_name` 缺失比例高，缺失本身需要保留为单独类别。
- `sex` 缺失明显，保留为 `MISSING` 比删除样本更合理。

## 9. 附加单集信息观察

- `uuid` 唯一值数：`1,807,479`
- 共享 `uuid` 的比例约 `10.32%`
- `title` 形态更接近脱敏 token 串

结论：

- `uuid_episode_count` 能够表达内容身份复用程度。

## 10. EDA 对特征工程的直接影响

本次 EDA 直接导向了以下特征设计：

- 对 `scene_name`、`entrance_type`、`tab_name`、`sex` 使用缺失类别化处理
- 对 `age` 做原值保留、分桶和异常标记
- 对 `duration` 同时构造原值、对数值和分桶值
- 对 `uid`、`episode_id` 及其交叉项构造频次统计
- 对主表关键字段与高基数 ID 构造 OOF target encoding

## 11. 关于 target encoding 泄漏

如果在训练集全量上直接计算某个类别的标签均值，再回填到训练样本本身，就会发生泄漏。

例如：

- 某个 `episode_id` 只出现 1 次，且 `label=1`
- 全量计算得到 `te_episode_id = 1.0`

此时模型实际上已经直接看到了答案。

因此，本次特征工程在训练集上采用 OOF target encoding，在测试集上使用全训练集统计，从而控制标签泄漏风险。
